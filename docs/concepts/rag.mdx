# RAG

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种将信息检索与大语言模型生成相结合的技术。它不是直接让 LLM 凭记忆回答问题，而是先从知识库中检索相关内容，再把检索到的内容作为上下文提供给 LLM，让 LLM 基于这些资料生成答案。

## RAG 的必要性

### 大语言模型的三大局限

1. **知识截止**

   LLM 的训练数据有截止时间。比如 GPT-4 的知识可能截止到 2024 年初，它不知道之后发生的事情。如果你的公司今年发布了新产品，LLM 完全不知道。

2. **幻觉问题**

   当 LLM 不确定答案时，它不会说"我不知道"，而是会一本正经地编造看起来合理的信息。这在企业场景中是不可接受的——客服机器人不能给用户编造产品参数。

3. **领域知识缺失**

   通用 LLM 对企业内部知识一无所知。你的产品手册、技术文档、规章制度，它从来没见过。

### RAG 如何解决这些问题

| 问题 | 纯 LLM | RAG |
|------|--------|-----|
| 知识时效性 | 受训练数据限制 | 实时检索最新文档 |
| 准确性 | 可能幻觉 | 基于检索内容，可溯源 |
| 领域适配 | 需要微调 | 直接接入领域文档 |
| 成本 | 微调成本高 | 仅需向量存储和检索 |
| 可解释性 | 不知道答案从哪来 | 可以展示引用来源 |

## RAG 的基本流程

```mermaid
graph LR
    Query[用户提问] --> Embed[问题向量化]
    Embed --> Search[向量检索 Top-K]
    Search --> Context[组装上下文]
    Context --> LLM[LLM 生成答案]
    LLM --> Answer[返回回答]
```

1. **用户提问**："这个产品的保修期是多久？"
2. **问题向量化**：把问题通过一个 Embedding 模型变成向量（一组数字）
3. **向量检索**：在向量数据库中查找和用户问题最相似的文档片段（Top-K）
4. **组装上下文**：把检索到的片段拼接成 prompt
5. **LLM 生成**：LLM 基于检索到的资料生成答案

## RAG 在 Synapse 中的增强设计

基础 RAG 只是"检索 → 生成"。Synapse 在这个基础上增加了多个优化：

### Query 改写

用户的问题可能不够精确。比如用户问"怎么用？"，系统不知道"用"什么。Query 改写会尝试把问题改写成更精确的形式："这个产品的主要使用步骤是什么？"

但改写可能改得更差。所以 Synapse 加了**质量门禁**：用 embedding 余弦相似度比较改写前后的问题，如果相似度低于 0.8，就回退到原问题。

### 混合检索

向量检索擅长找"语义相似"的内容，但不擅长找"包含特定关键词"的内容。比如用户问"型号 XJ-2025 的规格"，如果文档里没有"型号 XJ-2025"这个完整词组，向量检索可能找不到。

Synapse 同时使用两种检索：
- **向量召回（Milvus）**：找语义相近的片段
- **关键词召回（MongoDB BM25）**：找包含关键词的片段

然后融合两种结果，兼顾语义理解和精确匹配。

### 引用溯源

每个检索到的片段都有一个编号。LLM 回答时会用 `[1]`、`[2]` 标注引用了哪个片段。用户可以点击编号查看原文。

这不仅增加了可信度，也方便用户验证答案的准确性。

### 聊天记忆

多轮对话中，用户可能会说"它"指代前面的某个产品。Synapse 把聊天历史（摘要 + 最近消息）也放入 prompt，让 LLM 理解上下文。

但不会放入全量历史（会超限），而是压缩摘要 + 最近 N 条。

## RAG vs 微调（Fine-tuning）

微调 LLM 替代 RAG 的局限

| 对比项 | RAG | 微调 |
|--------|-----|------|
| 知识更新 | 上传新文档即可 | 需要重新训练 |
| 成本 | 向量存储费用低 | 训练费用高 |
| 幻觉控制 | 基于检索内容，可控 | 仍然可能幻觉 |
| 可解释性 | 可展示引用来源 | 无法解释 |
| 适用场景 | 频繁更新的知识库 | 固定知识且需要深度推理 |

**结论**：对于企业知识库这种频繁更新、需要准确溯源的场景，RAG 是更好的选择。微调更适合让模型学习特定的推理风格或格式。
