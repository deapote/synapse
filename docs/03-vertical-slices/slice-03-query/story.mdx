# 切片3：问答检索 —— 用户故事

## 场景

用户在聊天框输入问题："这个产品的主要功能是什么？" 系统需要从知识库中找到相关文档片段，基于这些片段生成答案，并以打字机效果逐字显示。

## 涉及的角色

| 角色 | 职责 |
|------|------|
| **用户** | 输入问题 |
| **浏览器** | 显示流式回答、引用来源 |
| **StreamingQueryController** | 接收 SSE 请求，推送事件 |
| **QueryKnowledgeBaseUseCase** | 编排检索和生成 |
| **OllamaQueryRewriteAdapter** | 改写问题，质量校验 |
| **OllamaEmbeddingAdapter** | 问题向量化 |
| **MilvusVectorStoreAdapter** | 向量相似度检索 |
| **MongoChunkSearchIndexAdapter** | BM25 关键词检索 |
| **OllamaStreamingLlmAdapter** | 流式生成答案 |

## 预期行为

### 正常流程

1. 用户输入问题
2. 系统尝试改写问题（可选）
3. 问题向量化
4. 并行执行向量检索和关键词检索
5. 融合两种结果，取 Top-K
6. 组装 Prompt（检索片段 + 用户问题）
7. LLM 流式生成答案
8. 前端逐字显示，带引用来源

### 异常流程

1. 检索结果为空 → 诚实回答"根据现有资料无法回答"
2. LLM 生成失败 → 返回错误事件，前端显示错误信息
3. 用户断开连接 → 取消 LLM 生成，释放资源

## 为什么这个切片最复杂

问答检索是 RAG 系统的**核心链路**，涉及多个技术点：

- **SSE 流式**：后端生成一个字，前端立刻显示一个字
- **Query 改写 + 质量门禁**：优化问题，但不过度改写
- **混合检索并行**：向量检索和关键词检索同时执行
- **融合重排**：两种结果加权排序
- **Prompt 工程**：隔离用户问题和检索内容，防止 prompt 注入
- **引用校验**：确保 LLM 引用的来源真实存在

学完这个切片后，你会理解：
- SSE 流式输出的完整实现
- 为什么需要 Query 改写和质量门禁
- 混合检索的设计思想
- Prompt 怎么组装才安全
