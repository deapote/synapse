# 领域模型设计模式

Domain 层有一些重复出现的设计模式，本节横向比较这些模式。

## 实体 vs 值对象 —— 判断标准

| 维度 | 实体（Entity） | 值对象（Value Object） |
|------|---------------|----------------------|
| 身份标识 | 有唯一 ID，ID 不变对象就是同一个 | 没有 ID，所有字段相同就相等 |
| 可变性 | 状态可变（如 `Document.status`） | 不可变（Java `record` 或 final class） |
| 生命周期 | 有独立生命周期，可被创建、修改、删除 | 依附于实体，随实体一起存取 |
| 相等判断 | 按 ID 判断 | 按所有字段判断 |

### 项目中的例子

**实体**：`Document`、`UserAccount`、`ChatSession`、`IngestionJob`
- 有 `id` 字段，即使其他字段全变，只要 ID 相同就认为是同一个对象
- 有状态变化：`PENDING → PROCESSING → COMPLETED`

**值对象**：`KnowledgeBaseId`、`DocumentId`、`Query`、`ChunkReference`、`DocumentChunk`

```java
// KnowledgeBaseId.java —— 值对象，record 天然不可变
public record KnowledgeBaseId(String value) {
    public KnowledgeBaseId {
        if (value == null || value.isBlank()) {
            throw new DomainException("知识库标识不能为空或者空白");
        }
    }
}
```

```java
// ChunkReference.java —— 值对象
public record ChunkReference(
    String documentId,
    String documentName,
    int chunkIndex,
    String chunkText,
    float score,
    int startPosition,
    int endPosition
) {}
```

<Tip>
值对象用 `record` 的好处：编译器自动生成 `equals()`、`hashCode()`、`toString()`，且字段不可重新赋值。这正是值对象需要的语义。
</Tip>

## create() vs reconstruct() —— 使用场景

这是项目中最重要的工厂模式约定。

| 方法 | 用途 | 是否走业务校验 | 调用方 |
|------|------|-------------|--------|
| `create()` | 业务代码创建新对象 | **是**，走完整校验和默认值 | Application Service、领域服务 |
| `reconstruct()` | 仓储层从数据库还原对象 | **否**，直接还原所有字段 | Repository 实现 |

### create() 与 reconstruct() 的必要性

```java
// UserAccount.java
public static UserAccount create(String username, String displayName,
                                  String passwordHash, Set<RoleName> roles) {
    validateUsername(username);           // 业务校验
    if (passwordHash == null || passwordHash.isBlank()) {
        throw new DomainException("密码哈希不能为空");
    }
    Set<RoleName> safeRoles = roles == null || roles.isEmpty()
            ? EnumSet.of(RoleName.USER)   // 默认值
            : EnumSet.copyOf(roles);
    return new UserAccount(UserId.generate(), username.trim(), ...);
}

public static UserAccount reconstruct(UserId id, String username, ...) {
    // 直接还原，不做校验——数据库里的数据已经校验过了
    return new UserAccount(id, username, ...);
}
```

如果没有 `reconstruct()`，仓储层从 MongoDB 读取一个 `FAILED` 状态的 `Document` 再保存时，`create()` 会强制把它重置为 `PENDING`，这显然不对。

## private 构造函数的设计原因

所有领域实体的构造函数都是 `private` 的：

```java
public class Document {
    private Document(DocumentId id, KnowledgeBaseId knowledgeBaseId, ...) { ... }
    // 只能通过工厂方法创建
}
```

**原因**：
1. **强制走工厂方法**——不允许绕过 `create()` 或 `reconstruct()`
2. **封装创建逻辑**——默认值、校验、副作用都在工厂方法里统一管理
3. **防止框架反射误用**——Spring Data MongoDB 通过 `reconstruct()` 还原，不走构造函数

## 状态机模式 —— Map + transitionTo()

`Document` 的状态转换不是随意赋值，而是查表校验：

```java
private static final Map<DocumentStatus, Set<DocumentStatus>> VALID_TRANSITIONS = Map.of(
    DocumentStatus.PENDING,   EnumSet.of(DocumentStatus.PROCESSING),
    DocumentStatus.PROCESSING, EnumSet.of(DocumentStatus.COMPLETED, DocumentStatus.FAILED),
    DocumentStatus.FAILED,    EnumSet.of(DocumentStatus.PENDING),  // 支持重试
    DocumentStatus.COMPLETED, EnumSet.noneOf(DocumentStatus.class)  // 终态
);
```

`transitionTo()` 三步走：
1. **查表**——当前状态允许转换到目标状态吗？
2. **校验**——不允许就抛 `DomainException`
3. **执行 + 副作用**——更新状态、记录时间、清理失败原因

```java
public void transitionTo(DocumentStatus newStatus, String failureReason) {
    Set<DocumentStatus> validNext = VALID_TRANSITIONS.get(this.status);
    if (validNext == null || !validNext.contains(newStatus)) {
        throw new DomainException("无效的状态转换 " + this.status + "->" + newStatus);
    }
    this.status = newStatus;
    if (newStatus == DocumentStatus.PROCESSING) {
        this.processingStartAt = Instant.now();
        this.failureReason = null;
    }
    // ... 更多副作用
}
```

### 对比：不用状态机的问题

```java
// 错误做法——直接赋值，任何状态都能跳转到任何状态
doc.setStatus(DocumentStatus.COMPLETED);  // PENDING 直接跳到 COMPLETED？合法吗？
```

状态机把"哪些转换是合法的"从业务员的脑子里搬到代码里，变成可执行规则。

## 领域服务的使用场景

**不是每个类都要做成实体**。有些操作符合以下特征时，应该用**领域服务**：

1. **操作多个实体**——如删除知识库时要清理文档、向量、索引、任务
2. **纯算法，无状态**——如文本分块、评分计算
3. **不适合放在任何一个实体里**——因为逻辑不属于某个特定实体

### 项目中的领域服务

**`RecursiveChunkingStrategy`** —— 纯算法，无状态
- 输入一段文本，输出 `List<DocumentChunk>`
- 不持有任何业务状态，线程安全
- 构造函数接收配置参数，创建后不可变

**`KnowledgeBaseApplicationService`** —— 严格来说不是领域服务，是应用服务。但它内部的一些私有方法（如 `mergeAndRerank`）是纯算法，如果逻辑更复杂，可以提取为独立的领域服务。

### 反模式：把算法塞进实体

```java
// 不要这样做
public class Document {
    public List<DocumentChunk> splitContent(String text) {
        // 分块逻辑有 100 行...
    }
}
```

分块策略和 `Document` 本身的生命周期无关，强行塞进去会让实体变得臃肿。分块是"对文本做的操作"，不是"文档自己的行为"。
