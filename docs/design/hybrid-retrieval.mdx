# 混合检索与融合重排

Synapse 的问答检索不是单一方式，而是**向量召回 + 关键词召回并行执行，结果融合后重排，再经过时效过滤和多版本去重**。本节深入讲解这个设计的原理和实现。

## 单一检索的局限

### 向量检索的局限

- **优势**：理解语义，"GPU 和显卡"能匹配到
- **劣势**：对精确词匹配弱，稀有术语（产品型号、人名）容易漏

### 关键词检索的局限

- **优势**：精确匹配，产品型号、代码片段找得准
- **劣势**：不理解同义词，"电脑"搜不到"计算机"

### 互补性

| 查询类型 | 向量检索 | 关键词检索 |
|---------|---------|-----------|
| "GPU 温度过高怎么办" | 强 | 中 |
| "RTX 4090 驱动版本 552.22" | 弱 | 强 |
| "系统崩溃了怎么排查" | 强 | 中 |
| "错误码 0x80070005" | 弱 | 强 |

两者结合，覆盖更多场景。

## 实现 —— CompletableFuture 并行

```java
private List<ChunkReference> retrieveReferences(Query query, PreparedQuery preparedQuery,
                                                  LocalDate asOfDate) {
    RetrievalFilter filter = new RetrievalFilter(asOfDate, null, null);

    CompletableFuture<List<ChunkReference>> vectorFuture = CompletableFuture.supplyAsync(
        () -> vectorStorePort.search(query.knowledgeBaseId(), preparedQuery.embedding(),
                                      vectorCandidateK, filter),
        retrievalExecutor
    );
    CompletableFuture<List<ChunkReference>> keywordFuture = CompletableFuture.supplyAsync(
        () -> chunkSearchIndexPort.search(query.knowledgeBaseId(), preparedQuery.effectiveText(),
                                           keywordCandidateK, filter),
        retrievalExecutor
    );
    List<ChunkReference> merged = mergeAndRerank(await(vectorFuture), await(keywordFuture));
    List<ChunkReference> filtered = filterByDocumentEffectiveDate(merged, asOfDate);
    return deduplicateByCanonicalKey(filtered);
}
```

**关键点**：
- 两个检索**并行**执行，`retrievalExecutor` 是虚拟线程池
- `vectorCandidateK` 和 `keywordCandidateK` 是候选集大小（默认各 20）
- `await()` 等待两个 Future 都完成，任一失败会抛异常
- 检索时都传入 `RetrievalFilter`，执行**时效硬过滤**

## 资料时效性不是重排加权，而是硬过滤

资料时效性的处理方式是**硬过滤**，不是重排时的分数加权或时间衰减。

**原因**：
- 过期法规/政策绝对不能被 LLM 引用，即使语义匹配度再高
- 时间衰减只能降低分数，不能保证过期资料被排除
- 硬过滤在检索阶段直接排除无效资料，是最安全的做法

**三层过滤**：
1. **Milvus scalar filter**：`effectiveFromEpochDay <= asOfDate && (effectiveToEpochDay == MAX || effectiveToEpochDay > asOfDate) && lifecycleStatus in [ACTIVE, SUPERSEDED]`
2. **Mongo BM25 Criteria**：同样的条件在 postings 查询中执行
3. **Application 兜底**：`filterByDocumentEffectiveDate()` 用 Mongo `Document` 真实状态二次校验

## 融合重排算法 —— 加权排序

```java
private List<ChunkReference> mergeAndRerank(
        List<ChunkReference> vectorResults,
        List<ChunkReference> keywordResults) {

    Map<String, RankedReference> merged = new LinkedHashMap<>();

    // 第一轮：放入向量结果，记录 vectorScore
    for (ChunkReference ref : vectorResults) {
        merged.computeIfAbsent(referenceKey(ref), k -> new RankedReference(ref))
              .vectorScore = ref.score();
    }

    // 第二轮：放入关键词结果，记录 keywordScore
    for (ChunkReference ref : keywordResults) {
        merged.computeIfAbsent(referenceKey(ref), k -> new RankedReference(ref))
              .keywordScore = ref.score();
    }

    // 加权融合 → 排序
    return merged.values().stream()
        .map(this::toFinalReference)
        .sorted(Comparator.comparing(ChunkReference::score).reversed())
        .toList();
}
```

### 唯一键 —— documentId + chunkIndex

```java
private String referenceKey(ChunkReference reference) {
    return reference.documentId() + ":" + reference.chunkIndex();
}
```

同一个文档的同一段内容，可能同时被向量检索和关键词检索命中，需要合并。

### 加权公式

```java
private ChunkReference toFinalReference(RankedReference ranked) {
    float score = (float) Math.min(1.0,
        vectorWeight * ranked.vectorScore + keywordWeight * ranked.keywordScore);
    return new ChunkReference(
        ref.documentId(), ref.documentName(), ref.chunkIndex(), ref.chunkText(),
        score, ref.startPosition(), ref.endPosition(),
        ref.canonicalKey(), ref.versionLabel(), ref.effectiveFrom(), ref.effectiveTo(),
        ref.lifecycleStatus(), ref.authorityLevel(), ref.jurisdiction()
    );
}
```

默认配置：
- `vectorWeight = 0.65`
- `keywordWeight = 0.35`

向量权重更高的原因：
- 向量检索的语义理解能力更贴近"用户真实意图"
- 关键词检索作为补充，提升精确术语的召回

### 权重归一化

构造函数里的安全处理：

```java
double safeVectorWeight = Math.max(0, vectorWeight);
double safeKeywordWeight = Math.max(0, keywordWeight);
double totalWeight = safeVectorWeight + safeKeywordWeight;
if (totalWeight == 0) {
    safeVectorWeight = 1;
    totalWeight = 1;
}
this.vectorWeight = safeVectorWeight / totalWeight;
this.keywordWeight = safeKeywordWeight / totalWeight;
```

即使配置写错了（如 `vectorWeight=0, keywordWeight=0`），也会自动归一化为 `1.0` 和 `0.0`，不会除以零。

## 同 canonicalKey 多版本去重

在 `mergeAndRerank` 之后、`limit(topK)` 之前，执行 `deduplicateByCanonicalKey()`：

```java
private List<ChunkReference> deduplicateByCanonicalKey(List<ChunkReference> ranked) {
    Map<String, ChunkReference> bestByCanonicalKey = new LinkedHashMap<>();
    List<ChunkReference> result = new ArrayList<>();

    for (ChunkReference ref : ranked) {
        String key = ref.canonicalKey();
        if (key == null || key.isBlank()) {
            result.add(ref);
            continue;
        }
        ChunkReference existing = bestByCanonicalKey.get(key);
        if (existing == null) {
            bestByCanonicalKey.put(key, ref);
            result.add(ref);
        } else {
            // authorityLevel 高者优先；同等级 effectiveFrom 更新者优先
            if (ref.authorityLevel() > existing.authorityLevel()
                || (ref.authorityLevel() == existing.authorityLevel()
                    && ref.effectiveFrom() != null && existing.effectiveFrom() != null
                    && ref.effectiveFrom().isAfter(existing.effectiveFrom()))) {
                bestByCanonicalKey.put(key, ref);
                int idx = result.indexOf(existing);
                if (idx >= 0) result.set(idx, ref);
            }
        }
    }
    return result.stream().limit(topK).toList();
}
```

**去重规则**：
- 同一 `canonicalKey` 只保留一个版本
- `authorityLevel` 高者优先（适用于不同层级法规的冲突）
- 同等级时，`effectiveFrom` 更新者优先（新法规替代旧法规）

**为什么必须在 topK 截断前去重？**

如果先 `limit(topK)` 再去重，可能把高优先级的版本截掉，留下低优先级版本。先去重再截断，保证最终返回的是最优版本集合。

## 加权融合替代 RRF

业界常用的另一种融合算法是 **RRF**：

```
RRF score = 1 / (k + rank_vector) + 1 / (k + rank_keyword)
```

RRF 的优点是不依赖原始分数的绝对值，只用排名。但 Synapse 没有用它，原因：

1. **Milvus 和 BM25 的分数都在 [0, 1] 区间**，可以直接加权
2. **RRF 对参数 k 敏感**，需要调参
3. **加权融合更直观**，开发者一眼看懂 `0.65 * vector + 0.35 * keyword`

如果未来接入外部搜索引擎（分数范围不一致），可以考虑切换到 RRF。

## 知识库隔离 —— 所有召回都带 knowledgeBaseId

```java
vectorStorePort.search(query.knowledgeBaseId(), preparedQuery.embedding(), vectorCandidateK, filter);
chunkSearchIndexPort.search(query.knowledgeBaseId(), preparedQuery.effectiveText(), keywordCandidateK, filter);
```

这是安全红线：**禁止跨知识库检索**。即使两个知识库有相同内容，也绝不能互相泄露。

Milvus 的 collection 里用 `knowledgeBaseId` 作为 partition key，MongoDB 的 postings 集合用 `knowledgeBaseId` 作为查询条件。
