# 前端架构设计

Synapse 前端使用 Vue 3 + Vite + Pinia。本节讲解后端开发者需要理解的前端核心设计，重点在状态管理、SSE 流式处理和资料时效性的前端实现。

## Pinia Store 设计模式

项目中每个 Store 都是 **Composition API 风格**：

```typescript
export const useChatStore = defineStore('chat', () => {
  // State
  const messages = ref<ChatMessage[]>([])
  const loading = ref(false)
  // ...

  // Actions
  function sendQuestionStream(knowledgeBaseId: string, query: string, asOfDate?: string) { ... }
  function stopGeneration() { ... }

  return { messages, loading, sendQuestionStream, stopGeneration }
})
```

对比 Options API 风格：
- **Composition API**：`ref` + 函数，和 Vue 3 `<script setup>` 一致
- 更灵活，逻辑可按功能组织而非按选项（state/mutations/actions）分割

## chat.ts 核心设计

### 竞态防护 —— currentGeneration

```typescript
let currentGeneration = 0

function sendQuestionStream(knowledgeBaseId: string, query: string, asOfDate?: string) {
    const generation = ++currentGeneration  // 每次请求递增

    // 添加用户消息和空助手消息
    // ...

    api.streamQueryKnowledgeBase(knowledgeBaseId, query, sessionId.value, {
        onToken: (token: string) => {
            if (generation !== currentGeneration) return  // 过期回调直接丢弃
            tokenBuffer += token
            startFlushTimer()
        },
        // ... 其他回调同理
    }, asOfDate)
}

function stopGeneration() {
    currentGeneration++  // 递增后，所有旧回调都被丢弃
    if (currentAbortCtrl) {
        currentAbortCtrl.abort()
    }
}
```

**问题场景**：用户快速连点发送、或切换知识库时旧请求才返回。如果不丢弃过期回调，消息会串到当前对话里。

**解决方案**：每次操作（发送、切换、新建会话）都递增 `currentGeneration`，回调里检查 generation 是否匹配。

### Token 缓冲 —— 解决 Vue 批量更新问题

```typescript
let tokenBuffer = ''
let flushTimer: ReturnType<typeof setInterval> | null = null

function startFlushTimer() {
    if (flushTimer) return
    flushTimer = setInterval(() => {
        flushTokens()
    }, 40)  // 每 40ms 刷新一次
}

function flushTokens() {
    if (!tokenBuffer) return
    const lastMsg = messages.value[messages.value.length - 1]
    if (lastMsg?.role === 'assistant') {
        lastMsg.content += tokenBuffer
    }
    tokenBuffer = ''
}
```

**问题**：SSE 的 `onToken` 回调每秒触发几十次，每次直接 `msg.content += token` 会导致 Vue 响应式系统频繁更新 DOM，打字机效果卡顿。

**解决方案**：token 先写入 `tokenBuffer`，定时器每 40ms 批量刷新一次。这样 DOM 更新频率从"每秒几十次"降到"每秒 25 次"，打字机效果流畅。

### AbortController —— 取消流式生成

```typescript
let currentAbortCtrl: AbortController | null = null

function sendQuestionStream(...) {
    currentAbortCtrl = api.streamQueryKnowledgeBase(..., { ... }, asOfDate)
}

function stopGeneration() {
    if (currentAbortCtrl) {
        currentAbortCtrl.abort()  // 触发 fetch signal 取消
        currentAbortCtrl = null
    }
}
```

`AbortController` 是浏览器原生 API，把 `signal` 传给 `fetch()` 后，调用 `abort()` 会中断网络连接，后端感知到连接断开会自动停止 LLM 生成。

## SSE 手动解析

`api/query.ts` 使用原生 `fetch + ReadableStream` 手动解析 SSE，而不是浏览器内置的 `EventSource`：

```typescript
const body: Record<string, string> = { query }
if (sessionId) body.sessionId = sessionId
if (asOfDate) body.asOfDate = asOfDate

fetch(`${baseUrl}/knowledge-bases/${knowledgeBaseId}/query/stream`, {
    method: 'POST',
    headers,
    body: JSON.stringify(body),
    signal: ctrl.signal,
})
```

**不用 EventSource 的原因**：
1. **无法自定义请求头**——EventSource 不支持 `Authorization` 或自定义 token header
2. **无法 POST 请求体**——EventSource 只能用 GET
3. **无法取消**——没有原生取消机制

手动解析的缺点是多写几十行代码，但获得了完全的控制权。

## 轮询文档状态 —— usePolling

```typescript
export function usePolling<T>(
  fn: () => Promise<T>,
  interval: number = 2000,   // 默认 2 秒
  maxAttempts: number = 30   // 最多 60 秒
) {
  function start(condition?: (result: T) => boolean) {
    timer = setInterval(async () => {
      attempts.value++
      if (attempts.value >= maxAttempts) { stop(); return }

      try {
        const result = await fn()
        if (condition?.(result)) { stop() }
      } catch {
        // ignore polling errors
      }
    }, interval)
  }

  onUnmounted(stop)  // 组件卸载自动停止

  return { isPolling, attempts, start, stop }
}
```

在 `KnowledgeBaseDetail.vue` 中使用：

```typescript
const polling = usePolling(fetchDocuments, 2000, 30)

// 开始轮询，直到所有文档都不在 PENDING/PROCESSING 状态
polling.start(docs => docs.every(d => d.status !== 'PENDING' && d.status !== 'PROCESSING'))
```

**WebSocket 的局限**
- 轮询实现简单，不需要维护 WebSocket 连接
- 文档状态变化不频繁（秒级），轮询足够
- 后端不需要额外的 WebSocket 基础设施

### 文档上传弹窗

```

│ [普通资料 ▼]                │

│ 规范标识      │ 版本标签    │
│ [law-x    ]   │ [2024 版  ] │
│                             │
│ 生效日期 *    │ 结束日期    │
│ [2024-01-01]  │ [2025-01-01]│
│               │ 到达该日期即失效 │
│                             │
│ 权威等级      │ 适用区域    │

│                             │

├─────────────────────────────┤
```

**表单校验规则**：
- 文件必填
- `sourceType` 必填，默认 `GENERAL`
- `effectiveFrom` 必填，默认今天
- `authorityLevel` 必须是整数，默认 0
- 选择 `supersedesDocumentId` 时，`effectiveFrom` 必填
- 空字符串字段不传递给后端
- **v2**：`sourceType` 和 `effectiveFrom` 不提供清空入口

**替代旧版本下拉**：只显示当前知识库 `status === 'COMPLETED' && lifecycleStatus !== 'RETIRED'` 的文档。

### 文档列表 —— 摄入状态、时效状态、索引状态分开展示（v2）

表格展示：文件名、大小、类型、时效状态、索引状态、生效期、摄入状态、操作。

文件名下方用 tag 展示 `versionLabel`、`canonicalKey`、`jurisdiction`：

```
法规A.pdf
├─ versionLabel: 2024 版
├─ canonicalKey: law-x
└─ jurisdiction: 全国
```

时效状态用三色 badge 区分：
- `ACTIVE`（绿色）—— 生效中
- `SUPERSEDED`（橙色）—— 已替代
- `RETIRED`（红色）—— 已废止

索引状态（v2）用额外标识：
- `SYNCED`（绿色圆点）—— 索引已同步
- `STALE`（黄色圆点）—— 索引待刷新
- `REFRESHING`（蓝色旋转图标）—— 刷新中
- `FAILED`（红色感叹号）—— 刷新失败

生效期展示体现排他语义：
- `effectiveTo` 为空：`2024-01-01 起长期有效`
- `effectiveTo` 有值：`2024-01-01 ~ 2025-01-01 前有效`

### 文档列表筛选（v2）

筛选栏提供四个下拉筛选器：
- `sourceType`：全部 / LEGAL / POLICY / GENERAL
- `lifecycleStatus`：全部 / ACTIVE / SUPERSEDED / RETIRED
- `indexStatus`：全部 / SYNCED / STALE / REFRESHING / FAILED
- `canonicalKey`：文本输入框，精确匹配

### 元数据编辑弹窗（v2）

`DocumentMetadataEditDialog.vue`：
- `sourceType`：下拉选择（GENERAL/LEGAL/POLICY），**不提供清空**
- `effectiveFrom`：日期选择器，**不提供清空**
- 其他字段：输入框 + "清空"复选框（支持 `clear` 语义）
- 保存时只发送变更字段（`unset` 的字段不发送）

保存后：
1. 调用 `PATCH /api/documents/{id}/metadata`
2. 抽屉中 `indexStatus` 变为 `STALE`
3. 开始轮询索引状态，直到 `SYNCED` 或 `FAILED`

### 替代弹窗（v2）

`DocumentSupersedeDialog.vue`：
- 选择新文档：下拉列表，候选为同一知识库 `COMPLETED` 状态的文档
- 填 `effectiveTo`：日期选择器
- 被替代文档必须是 `ACTIVE` 状态

### 生命周期操作（v2）

`DocumentLifecycleActions.vue`：
- **废止**（retire）：`ACTIVE` 文档可用，弹出确认框
- **重新启用**（reactivate）：`RETIRED` 文档可用
- **刷新索引**（reindex）：任何状态可用，用于 `FAILED` 恢复

操作后同步更新抽屉中的文档状态和索引状态。

### 索引状态轮询（v2）

reindex 操作后启动独立轮询：

```typescript
async function handleReindex(docId: string) {
  await docStore.reindexDoc(docId)
  // 启动轮询，直到索引状态变为 SYNCED 或 FAILED
  const poll = setInterval(async () => {
    const doc = await docStore.fetchSingle(docId)
    if (doc.indexStatus === 'SYNCED' || doc.indexStatus === 'FAILED') {
      clearInterval(poll)
      selectedDoc.value = doc
    }
  }, 3000)
}
```

## 设计约束

1. **不引入新 UI 框架**——所有弹窗、表单、badge 使用现有 CSS 变量和组件风格实现
2. **不破坏旧流程兼容**——不传 metadata 时 upload API 仍然工作
3. **不混用两个状态**——`Document.status`（摄入状态）和 `lifecycleStatus`（时效状态）在前端 UI 中始终分开展示
4. **不做前端过期判断**——资料是否过期以后端检索结果为准，前端只展示
5. **v2 治理操作后同步 UI**——metadata 编辑、supersede、retire、reactivate、reindex 后，立即更新本地文档状态和索引状态，并启动轮询观察后台进度
