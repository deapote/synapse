# 端口设计模式

端口（Port）是六边形架构的核心概念。端口的命名和位置遵循严格的规律。

## 两种端口

```
           Driving Port (入站)              Driven Port (出站/SPI)
           ───────────────────              ──────────────────────

   外部调用方 ──→ Application 层 ──→ 端口接口 ──→ Adapter 实现 ──→ 外部服务
   (Controller)     (UseCase)         (Port)       (Adapter)      (Mongo/Ollama)
```

| 类型 | 命名 | 包位置 | 谁实现 | 谁调用 |
|------|------|--------|--------|--------|
| Driving Port | `*UseCase` | `port/in/` | Application Service | Controller / Worker |
| Driven Port | `*Port` | `port/out/` | Adapter | Application Service |

## 项目中的端口清单

### Auth Context

**入站端口（Driving）**：
- `AuthenticationUseCase` —— 登录、登出、当前用户
- `UserAdminUseCase` —— 用户/角色的增删改查

**出站端口（Driven）**：
- `LoginSessionPort` —— 会话管理抽象
- `PasswordHasherPort` —— 密码哈希抽象

### KB Context

**入站端口（Driving）**：
- `CreateKnowledgeBaseUseCase`、`DeleteKnowledgeBaseUseCase`、`ListKnowledgeBaseUseCase`
- `IngestDocumentUseCase`、`ListDocumentUseCase`、`DeleteDocumentUseCase`、`RetryDocumentIngestionUseCase`
- `ProcessIngestionJobUseCase`
- `QueryKnowledgeBaseUseCase`
- `GetCurrentChatSessionUseCase`、`CreateChatSessionUseCase`、`ListChatMessagesUseCase`
- `UpdateDocumentMetadataUseCase` —— 在线修改文档元数据（v2）
- `SupersedeDocumentUseCase` —— 手动建立版本替代关系（v2）
- `RetireDocumentUseCase` —— 废止文档（v2）
- `ReactivateDocumentUseCase` —— 重新启用已废止文档（v2）
- `ReindexDocumentUseCase` —— 强制重建索引（v2）
- `GetDocumentVersionChainUseCase` —— 查询文档版本链（v2）
- `GetDocumentAuditEventsUseCase` —— 查询审计事件（v2）

**出站端口（Driven）**：
- `VectorStorePort` —— 向量存储（Milvus v3）
- `ChunkSearchIndexPort` —— 关键词索引（MongoDB）

- `QueryRewritePort` —— Query 改写（Ollama）

## DocumentParserPort 和解析 Provider

文件：`synapse-kb-application/.../port/out/DocumentParserPort.java`

```java
public interface DocumentParserPort {
    String parse(InputStream inputStream, String fileName);
}
```

`DocumentParserPort` 是文档摄入链路里唯一的解析抽象。Application 层不知道当前使用 MinerU 还是 Tika，只接收解析后的纯文本或 Markdown 文本，再交给 `RecursiveChunkingStrategy` 分块。

当前 adapter 实现：

| provider | 实现类 | 外部依赖 | 适用场景 |
|----------|--------|----------|----------|
| `mineru` | `MineruDocumentParserAdapter` | 独立 `mineru-api` 服务 | PDF 版面、表格、公式解析质量优先 |
| `tika` | `ApacheTikaDocumentParserAdapter` | Java 进程内 Tika | 轻量 fallback、纯文本/Office 基础解析 |

`synapse.parser.provider=mineru` 时，`MineruDocumentParserAdapter` 通过 REST multipart 调用 `/file_parse`，并在 `fallback-to-tika=true` 时复用 Tika 做兜底。这个选择保持了端口边界：Python 环境、pipeline 模型和模型缓存都在 Synapse Java 进程之外。

### RetrievalFilter

文件：`synapse-kb-application/.../port/out/RetrievalFilter.java`

```java
public record RetrievalFilter(
    LocalDate asOfDate,           // 必填硬过滤
    DocumentSourceType sourceType, // 可选扩展过滤
    String jurisdiction            // 可选扩展过滤
) {
    public RetrievalFilter {
        if (asOfDate == null) {
            throw new IllegalArgumentException("asOfDate 不能为空");
        }
    }
}
```

`RetrievalFilter` 是检索时的过滤条件：
- `asOfDate` 为**必填**硬过滤，Milvus 和 Mongo 检索都必须执行
- `sourceType` 和 `jurisdiction` 为可选软过滤，v1 中索引已支持，可在后续版本开放给用户

### UpdateDocumentMetadataUseCase / SupersedeDocumentUseCase

文件：`synapse-kb-application/.../port/in/UpdateDocumentMetadataUseCase.java`

```java
public interface UpdateDocumentMetadataUseCase {
    Document updateMetadata(DocumentId id, PatchDocumentMetadata patch);
}
```

文件：`synapse-kb-application/.../port/in/SupersedeDocumentUseCase.java`

```java
public interface SupersedeDocumentUseCase {
    void supersede(DocumentId oldDocumentId, DocumentId newDocumentId, LocalDate effectiveTo);
}
```

v2 中这些接口由多个 Application Service 分别实现：

- **`DocumentGovernanceService`**：实现 `UpdateDocumentMetadataUseCase`、`SupersedeDocumentUseCase`、`RetireDocumentUseCase`、`ReactivateDocumentUseCase`、`ReindexDocumentUseCase`、`GetDocumentVersionChainUseCase`、`GetDocumentAuditEventsUseCase`
  - `updateMetadata`：调用 `document.patchMetadata(patch)` 做原子校验和赋值，然后 `markIndexStale()` 触发后台索引刷新
  - `supersede`：校验旧文档为 `ACTIVE`、新旧文档同属一个知识库，然后 `oldDoc.supersedeBy(newDoc, effectiveTo)`
  - `retire` / `reactivate`：变更 `lifecycleStatus`，标记索引 `STALE`
  - `reindex`：强制递增 `metadataVersion` 并创建刷新任务
  - `getVersionChain`：基于 `supersedesDocumentId` 向前/向后追踪构建版本链
  - `getAuditEvents`：查询 `document_audit_events` 集合

## 接口隔离的设计原因

### 1. 测试时可 Mock

```java
@Test
void shouldRejectDuplicateContent() {
    DocumentRepository mockRepo = mock(DocumentRepository.class);
    when(mockRepo.findByKnowledgeBaseIdAndContentHash(any(), any()))
        .thenReturn(List.of(existingDoc));

    DocumentCommandService service = new DocumentCommandService(
        mockRepo, ...  // 其他端口也 mock
    );

    assertThrows(DomainException.class, () -> service.ingest(command));
}
```

### 2. 实现可替换

今天用 Ollama 做 Embedding，明天可以换成 OpenAI，只需要：
1. 新建 `OpenAiEmbeddingAdapter implements EmbeddingPort`
2. 修改 Config 层的 Bean 装配

Application Service 一行代码不用改。

### 3. 依赖方向可控

```
Domain / Application 层只 import 端口接口
    ↓
完全不依赖 Spring、MongoDB、LangChain4j
    ↓
技术栈变更时，上层代码不受影响
```

## 端口粒度 —— 一个场景一个接口

不要搞"大接口"：

```java
// 错误做法——所有操作塞在一个接口里
public interface KnowledgeBasePort {
    void createKB(...);
    void deleteKB(...);
    void ingestDoc(...);
    List<ChunkReference> search(...);
    // ... 30 个方法
}
```

正确做法——按**用例**拆分：

```java
public interface CreateKnowledgeBaseUseCase {
    KnowledgeBaseId create(CreateKnowledgeBaseCommand command);
}

public interface IngestDocumentUseCase {
    DocumentId ingest(IngestDocumentCommand command);
}

public interface QueryKnowledgeBaseUseCase {
    RagContext prepare(Query query);
    void complete(RagContext ragContext, String answerText);
}
```

每个接口只负责一个**完整的用户场景**，调用方（Controller）按需依赖，不用被迫了解无关方法。

## Application Service 实现多个 UseCase

```java
public class KnowledgeBaseManagementService implements
        CreateKnowledgeBaseUseCase,
        DeleteKnowledgeBaseUseCase,
        ListKnowledgeBaseUseCase { }

public class DocumentCommandService implements
        IngestDocumentUseCase,
        DeleteDocumentUseCase,
        RetryDocumentIngestionUseCase { }

public class DocumentIngestionService implements
        ProcessIngestionJobUseCase { }

public class DocumentGovernanceService implements
        UpdateDocumentMetadataUseCase,
        SupersedeDocumentUseCase,
        RetireDocumentUseCase,
        ReactivateDocumentUseCase,
        ReindexDocumentUseCase,
        GetDocumentVersionChainUseCase,
        GetDocumentAuditEventsUseCase { }

public class QueryKnowledgeBaseApplicationService implements
        QueryKnowledgeBaseUseCase { }

public class ChatApplicationService implements
        GetCurrentChatSessionUseCase,
        CreateChatSessionUseCase,
        ListChatMessagesUseCase { }
```

这是项目中的模式：按职责将 Driving Port 拆分到多个 Application Service 中。好处：
- 端口粒度细（按用例），类粒度也适中（按职责域）
- 避免单类膨胀（原 `KnowledgeBaseApplicationService` 超过 1300 行）
- 共享私有方法提取到 `support/` 包下的辅助类（如 `KnowledgeBaseAccessGuard`、`FailureReasonSanitizer`）

## AccessControlPort —— 跨上下文 ACL

`kb-application` 不允许直接依赖 `auth-adapter`（违反依赖方向），鉴权能力通过端口暴露：

```java
// kb-application 定义端口
public interface AccessControlPort {
    String currentUserId();
    boolean isAdmin();
    void checkPermission(String permission);
    void checkKnowledgeBaseAccess(KnowledgeBase knowledgeBase, String permission);
}
```

```java
// auth-adapter 实现端口（在 auth 上下文里）
@Component
public class SaTokenAccessControlAdapter implements AccessControlPort {
    // 内部调用 StpUtil
}
```

Config 层把 `SaTokenAccessControlAdapter` 注入给各 Application Service（如 `DocumentCommandService`、`QueryKnowledgeBaseApplicationService` 等）。

这就是**防腐层（Anti-Corruption Layer, ACL）**模式：KB 上下文用自己的语言定义"我需要什么权限能力"，不关心 Sa-Token 的具体 API。
