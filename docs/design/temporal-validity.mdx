# 资料时效性治理

## 问题背景

在法规、政策、制度类知识库场景中，同一份资料往往存在多个历史版本：

- 法规 A 于 2024-01-01 生效
- 法规 A 修订版于 2025-01-01 生效，旧版同时废止
- 用户问 "2024 年的规定是什么？" 时，系统必须返回 2024 版，而非 2025 版

传统的 RAG 系统没有版本和时效概念，检索时可能同时命中新旧版本，导致 LLM 引用错误资料给出过时回答。

Synapse 的资料时效性治理通过以下机制解决这个问题：
1. 文档携带 `effectiveFrom`、`effectiveTo`、`lifecycleStatus`、`canonicalKey` 等时效元数据
2. 检索时按 `asOfDate` 在 Milvus 和 Mongo 双侧执行硬过滤
3. Application 层用 Mongo `Document` 真实状态兜底过滤
4. 同 `canonicalKey` 多版本去重，只保留最优版本
5. **v2** 提供完整的在线治理能力：metadata patch、手动 supersede、retire/reactivate、强制 reindex、索引状态追踪、审计日志

## 领域模型

### Document 与时效相关字段

```java
public class Document {
    // 摄入状态（与时效状态严格分离）
    private DocumentStatus status;  // PENDING/PROCESSING/COMPLETED/FAILED

    // 时效元数据
    private DocumentSourceType sourceType;           // LEGAL/POLICY/GENERAL
    private String canonicalKey;                     // 多版本关联键
    private String versionLabel;                     // 版本标签
    private LocalDate effectiveFrom;                 // 生效日期（必填）
    private LocalDate effectiveTo;                   // 排他结束日期
    private DocumentLifecycleStatus lifecycleStatus; // ACTIVE/SUPERSEDED/RETIRED
    private String supersedesDocumentId;             // 被替代的旧文档 ID
    private int authorityLevel;                      // 权威等级
    private String jurisdiction;                     // 适用区域

    // v2: 索引同步状态
    private long metadataVersion;           // 每次 metadata 变更递增
    private long indexedMetadataVersion;    // 当前索引对应的 metadata 版本
    private DocumentIndexStatus indexStatus; // SYNCED/STALE/REFRESHING/FAILED
    private Instant lastIndexRefreshAt;
    private String lastIndexFailureReason;
}
```

### DocumentMetadata —— 值对象

```java
public record DocumentMetadata(
    DocumentSourceType sourceType,
    String canonicalKey,
    String versionLabel,
    LocalDate effectiveFrom,
    LocalDate effectiveTo,
    String supersedesDocumentId,
    Integer authorityLevel,
    String jurisdiction
) {}
```

- `sourceType` 默认 `GENERAL`
- `effectiveFrom` 默认上传日期，**必填且不允许清空**
- `lifecycleStatus` 默认 `ACTIVE`
- `authorityLevel` 默认 `0`

### DocumentLifecycleStatus

```java
public enum DocumentLifecycleStatus {
    ACTIVE,      // 生效中，可被检索
    SUPERSEDED,  // 已被新版本替代，但在 asOfDate 之前仍有效
    RETIRED      // 已废止，任何 asOfDate 都不可检索
}
```

### DocumentIndexStatus

```java
public enum DocumentIndexStatus {
    SYNCED,      // 索引与 Mongo 权威状态一致
    STALE,       // Mongo 状态已变，索引待刷新
    REFRESHING,  // 索引刷新任务正在执行
    FAILED       // 索引刷新失败，需人工干预
}
```

三个状态维度严格分离：
- `DocumentStatus` 管**摄入过程**
- `DocumentLifecycleStatus` 管**业务时效**
- `DocumentIndexStatus` 管**索引一致性**

### effectiveTo 的排他语义

`effectiveTo` 是**排他结束日**：

| effectiveTo | 在 asOfDate 下是否有效 |
|-------------|----------------------|
| null | 永远有效 |
| 2025-01-01 | asOfDate < 2025-01-01 时有效；asOfDate >= 2025-01-01 时无效 |

这意味着新版本的 `effectiveFrom` 等于旧版本的 `effectiveTo` 时，两者之间没有重叠，也没有间隙。

### PatchValue<T> —— v2 在线编辑语义

```java
public final class PatchValue<T> {
    public static <T> PatchValue<T> unset() { ... }  // 不修改
    public static <T> PatchValue<T> set(T value) { ... }  // 覆盖
    public static <T> PatchValue<T> clear() { ... }  // 显式清空
}
```

在线编辑必须区分"不传"和"传 null"。`sourceType` 和 `effectiveFrom` 不允许 `clear`，其余字段可 `clear`。

## 摄入版本链

### 上传第一版

```bash
curl -X POST "http://localhost:8082/api/knowledge-bases/{kbId}/documents" \
  -H "synapse-token: xxx" \
  -F "file=@law-2024.pdf" \

  -F "versionLabel=2024 版" \

  -F "sourceType=LEGAL"
```

创建的 Document：
- `status=PENDING` → `PROCESSING` → `COMPLETED`
- `lifecycleStatus=ACTIVE`
- `effectiveFrom=2024-01-01`
- `effectiveTo=null`

### 上传第二版（替代第一版）

```bash
curl -X POST "http://localhost:8082/api/knowledge-bases/{kbId}/documents" \
  -H "synapse-token: xxx" \
  -F "file=@law-2025.pdf" \
  -F "canonicalKey=law-x" \
  -F "versionLabel=2025 版" \
  -F "effectiveFrom=2025-01-01" \
  -F "sourceType=LEGAL" \
  -F "supersedesDocumentId=<v1DocumentId>"
```

创建的 Document：
- `status=PENDING` → `PROCESSING` → `COMPLETED`
- `lifecycleStatus=ACTIVE`
- `effectiveFrom=2025-01-01`
- `supersedesDocumentId=<v1DocumentId>`

旧文档在第一版 COMPLETED 后被自动标记：
- `lifecycleStatus=SUPERSEDED`
- `effectiveTo=2025-01-01`（新版本的 effectiveFrom）

### 版本替代的关键设计

**旧文档只在新文档摄入成功后才被替代**。如果新版本摄入失败：
- 旧文档保持 `ACTIVE`
- 系统不会失去有效资料
- 用户可重试新版本上传

```java
private void supersedeOldDocument(Document newDocument) {
    DocumentId oldId = new DocumentId(newDocument.getSupersedesDocumentId());
    Document oldDoc = documentRepository.findById(oldId).orElse(null);
    if (oldDoc == null) {
        log.warn("被替代文档不存在，跳过替代");
        return;
    }

    oldDoc.supersedeBy(newDocument, newDocument.getEffectiveFrom());
    documentRepository.save(oldDoc);
}
```

## v2 在线治理能力

### 在线 metadata patch

`PUT /api/documents/{id}/metadata` 支持在线修改文档元数据。修改后：
1. `document.patchMetadata(patch)` 原子校验（先计算候选值，校验通过后统一赋值）
2. `document.markIndexStale()` → `metadataVersion++`，`indexStatus = STALE`
3. 创建 `DocumentIndexRefreshJob`
4. 写审计日志 `METADATA_UPDATED`

约束：
- `sourceType` 不允许 `clear`
- `effectiveFrom` 不允许 `clear`
- `effectiveTo <= effectiveFrom` 被拒绝
- `authorityLevel < 0` 被拒绝

### 手动 supersede

`POST /api/documents/{id}/supersede` 手动建立替代关系：
1. 校验旧文档 `lifecycleStatus == ACTIVE`
2. 校验新旧文档属于同一知识库
3. `oldDoc.supersedeBy(newDoc, effectiveTo)`
4. 标记索引 `STALE`，创建刷新任务
5. 写审计日志 `SUPERSEDED`

### retire / reactivate

`POST /api/documents/{id}/retire`：
- `document.retire()` → `lifecycleStatus = RETIRED`
- 标记索引 `STALE`，创建刷新任务
- 写审计日志 `RETIRED`

`POST /api/documents/{id}/reactivate`：
- `document.reactivate()` → `lifecycleStatus = ACTIVE`，清除 `effectiveTo`
- 标记索引 `STALE`，创建刷新任务
- 写审计日志 `REACTIVATED`

### 强制 reindex

`POST /api/documents/{id}/reindex`：
- 手动触发索引刷新
- `metadataVersion++`，标记 `STALE`
- 创建刷新任务
- 写审计日志 `INDEX_REFRESH_REQUESTED`

用于 `FAILED` 状态恢复或人工干预。

### 索引刷新任务

```java
public class DocumentIndexRefreshJob {
    private String id;
    private DocumentId documentId;
    private KnowledgeBaseId knowledgeBaseId;
    private long metadataVersion;  // 触发时的 metadata 版本
    private DocumentIndexRefreshJobStatus status;  // PENDING/RUNNING/SUCCEEDED/FAILED
    private int attempts;
    private String failureReason;
    private Instant nextRunAt;
}
```

Worker 调度：
1. claim 一个 `PENDING` job
2. 读取 Document，若 `metadataVersion != job.metadataVersion` 则跳过（旧 job）
3. `document.markIndexRefreshing()`
4. 从 `documentChunkRepository` 读取持久化 chunks
5. 重新生成 embeddings
6. `vectorStorePort.store()` 写入 Milvus（旧行保留）
7. `chunkSearchIndexPort.refreshStore()` 安全刷新 Mongo BM25
8. `document.markIndexSynced()`，`indexedMetadataVersion = metadataVersion`
9. job 标记 `SUCCEEDED`
10. 失败时：`document.markIndexFailed(reason)`，job 标记 `FAILED` 并设置重试时间

`GET /api/documents/{id}/audit-events` 返回：

## 检索过滤链路

### Query 与 asOfDate

```java
public record Query(

```

`asOfDate` 的解析逻辑：

```java
private LocalDate resolveAsOfDate(Query query) {
    if (query.asOfDate() != null) {
        return query.asOfDate();
    }
    LocalDate parsed = parseTemporalIntent(query.text());
    return parsed != null ? parsed : LocalDate.now();
}
```

轻量时间意图解析支持：
- `"2024年规定"` → `2024-12-31`
- `"现在适用"` → `LocalDate.now()`

### RetrievalFilter

```java
public record RetrievalFilter(
    LocalDate asOfDate,           // 必填硬过滤
    DocumentSourceType sourceType, // 可选扩展过滤

```

### 三层过滤

#### 第一层：Milvus scalar 硬过滤

```java
private String buildSearchFilter(KnowledgeBaseId knowledgeBaseId, RetrievalFilter filter) {
    long asOfEpochDay = filter.asOfDate().toEpochDay();
    return "knowledgeBaseId == '" + kbId + "'"
        + " && effectiveFromEpochDay <= " + asOfEpochDay
        + " && (effectiveToEpochDay == " + Long.MAX_VALUE
        + " || effectiveToEpochDay > " + asOfEpochDay + ")"
        + " && lifecycleStatus in ['ACTIVE', 'SUPERSEDED']";
}
```

#### 第二层：Mongo BM25 Criteria 硬过滤

```java
Criteria criteria = Criteria.where("knowledgeBaseId").is(knowledgeBaseId.value())
    .and("token").in(uniqueQueryTokens)
    .and("lifecycleStatus").in("ACTIVE", "SUPERSEDED")
    .and("effectiveFromEpochDay").lte(asOfEpochDay)
    .andOperator(new Criteria().orOperator(
        Criteria.where("effectiveToEpochDay").is(Long.MAX_VALUE),
        Criteria.where("effectiveToEpochDay").gt(asOfEpochDay)
    ));
```

#### 第三层：Application 兜底过滤（含 SYNCED 检查 + Mongo 权威 metadata 覆盖）

```java
private List<ChunkReference> filterByDocumentEffectiveDate(
        List<ChunkReference> references, LocalDate asOfDate) {
    Set<String> documentIds = references.stream()
        .map(ChunkReference::documentId).collect(Collectors.toSet());
    Map<String, Document> documentMap = new HashMap<>();
    for (String docId : documentIds) {
        documentRepository.findById(new DocumentId(docId))
            .ifPresent(doc -> documentMap.put(docId, doc));
    }
    return references.stream()
        .filter(ref -> {
            Document doc = documentMap.get(ref.documentId());
            if (doc == null) return true;
            // 排除索引不同步的文档
            if (doc.getIndexStatus() != DocumentIndexStatus.SYNCED) {
                return false;
            }
            // 用 Mongo 权威 metadata 覆盖索引中的时效字段
            return doc.isEffectiveOn(asOfDate);
        })
        .map(ref -> {
            Document doc = documentMap.get(ref.documentId());
            if (doc == null) return ref;
            return new ChunkReference(
                ref.documentId(), ref.documentName(), ref.chunkIndex(), ref.chunkText(),
                ref.score(), ref.startPosition(), ref.endPosition(),
                doc.getCanonicalKey(), doc.getVersionLabel(),
                doc.getEffectiveFrom(), doc.getEffectiveTo(),
                doc.getLifecycleStatus(), doc.getAuthorityLevel(), doc.getJurisdiction()
            );
        })
        .toList();
}
```

**为什么需要 SYNCED 检查？**

治理操作（patch metadata、supersede、retire、reactivate）会修改 Mongo 中的 `Document` 状态，但索引刷新是异步的。在刷新完成前，Milvus/Mongo 索引中仍然保留旧 metadata。通过排除 `indexStatus != SYNCED` 的文档，避免返回基于旧索引的不准确结果。

**为什么需要 Mongo 权威 metadata 覆盖？**

即使索引已刷新，Milvus 中的旧行可能仍然存在（因为刷新策略是"写入新行，不 delete 旧行"）。用 Mongo `Document` 的真实状态覆盖 `ChunkReference` 中的时效字段，确保引用展示的 metadata 100% 准确。

### 完整检索 pipeline

```
向量召回（Milvus v3，scalar 硬过滤）
    ↕ 并行
关键词召回（Mongo BM25，Criteria 硬过滤）
    ↓
mergeAndRerank（加权融合分数 [0, 1]）
    ↓
filterByDocumentEffectiveDate（Mongo Document 真实状态兜底 + SYNCED 检查 + 权威 metadata 覆盖）
    ↓
deduplicateByCanonicalKey（同 canonicalKey 多版本去重）
    ↓
limit(topK)
```

### 同 canonicalKey 多版本去重

```java
private List<ChunkReference> deduplicateByCanonicalKey(List<ChunkReference> ranked) {
    Map<String, ChunkReference> bestByCanonicalKey = new LinkedHashMap<>();
    List<ChunkReference> result = new ArrayList<>();

    for (ChunkReference ref : ranked) {
        String key = ref.canonicalKey();
        if (key == null || key.isBlank()) {
            result.add(ref);
            continue;
        }
        ChunkReference existing = bestByCanonicalKey.get(key);
        if (existing == null) {
            bestByCanonicalKey.put(key, ref);
            result.add(ref);
        } else {
            // authorityLevel 高者优先；同等级 effectiveFrom 更新者优先
            if (ref.authorityLevel() > existing.authorityLevel()
                || (ref.authorityLevel() == existing.authorityLevel()
                    && ref.effectiveFrom() != null && existing.effectiveFrom() != null
                    && ref.effectiveFrom().isAfter(existing.effectiveFrom()))) {
                bestByCanonicalKey.put(key, ref);
                int idx = result.indexOf(existing);
                if (idx >= 0) result.set(idx, ref);
            }
        }
    }
    return result.stream().limit(topK).toList();
}
```

**去重规则**：
- 同一 `canonicalKey` 只保留一个版本
- `authorityLevel` 高者优先（适用于不同层级法规的冲突）
- 同等级时，`effectiveFrom` 更新者优先（新法规替代旧法规）

**为什么必须在 topK 截断前去重？**

如果先 `limit(topK)` 再去重，可能把高优先级的版本截掉，留下低优先级版本。先去重再截断，保证最终返回的是最优版本集合。

## 索引一致性策略

### 核心不变量

> **Mongo `Document` 是权威状态源，Milvus/Mongo chunk index 是派生索引。**

所有治理操作只修改 Mongo 中的 `Document` 状态，不直接修改索引。索引通过异步刷新任务与 Mongo 状态保持一致。

### 索引刷新流程

```
governance operation (patch/supersede/retire/reactivate/reindex)
    ↓
Document.markIndexStale() → metadataVersion++
    ↓
DocumentRepository.save(document)
    ↓
DocumentIndexRefreshJobRepository.save(job)
    ↓
Worker.claimJob() → runIndexRefreshTask()
    ↓
1. markIndexRefreshing()
2. read persistent chunks from document_chunk_repository
3. regenerate embeddings
4. vectorStorePort.store() → Milvus (write new rows, keep old rows)
5. chunkSearchIndexPort.refreshStore() → Mongo BM25 (safe two-phase refresh)
6. markIndexSynced() → indexedMetadataVersion = metadataVersion
```

### Milvus 刷新策略

- **写入新行**：`vectorStorePort.store()` 为每个 chunk 生成新的 embedding 记录
- **不 delete 旧行**：旧行仍然存在于 collection 中
- **兜底过滤**：`filterByDocumentEffectiveDate()` 排除 `indexStatus != SYNCED` 的文档
- **权威覆盖**：即使命中了旧行，也用 Mongo `Document` 的真实 metadata 覆盖 `ChunkReference`

**为什么保留旧行？**

避免"delete 旧行 → 写入新行失败 → 索引完全丢失"的风险。旧行在检索阶段被 SYNCED 检查排除，不会污染结果。

**未来优化**：引入 generation/alias 机制或后台物理清理任务，定期清理旧行。

### Mongo BM25 安全刷新

`ChunkSearchIndexPort.refreshStore()` 实现两阶段写入：

1. **记录旧数据**：读取现有的 `ChunkIndexDocument` 列表（用于 corpus stats diff）
2. **识别 orphaned chunks**：新 chunks 中不再存在的旧 chunk index
3. **写入新 chunk index**：`saveAll` 覆盖同 ID 的记录
4. **删除 orphaned chunks**：`repository.deleteById()` 物理删除
5. **准备新 postings**：基于新 chunk index 生成
6. **获取旧 postings 的 _id 集合**
7. **保存新 postings**：`mongoTemplate.save()` upsert 覆盖同 ID
8. **按 _id 差集删除旧 postings**：精确删除不再存在的 posting，不依赖 token 集合
9. **更新 corpus stats**：diff = newTotal - oldTotal

**为什么不用 token 集合删除？**

如果旧 token 仍在新文档的其他 chunk 中出现，按 token 删除会误删新 chunk 的 posting。按 `_id` 差集删除精确到每个 posting，不会误删或漏删。

## 边界和约束

### 存量数据迁移

旧 collection `synapse_document_chunks`（v1/v2）不包含时效字段。v3 collection `synapse_document_chunks_v3` 拥有完整的时效 scalar 字段。

**存量数据必须重新摄入或迁移到 v3**，否则检索时无法执行 `effectiveFromEpochDay` / `effectiveToEpochDay` / `lifecycleStatus` 硬过滤，时效治理不完整。

### 设计约束

- `MilvusVectorStoreAdapter` 不得依赖 Mongo BM25 adapter/entity
- 不允许为了在线更新 metadata 对 Milvus 做无补偿 delete + reinsert
- 时效过滤必须在 `mergeAndRerank` 后、`limit(topK)` 前执行
- 禁止只用时间衰减/重排替代有效期硬过滤
- `sourceType` 和 `effectiveFrom` 不允许清空
- 删除文档时必须同时清理索引刷新任务（`refreshJobRepository.deleteByDocumentId`）
- 索引刷新失败不会自动回退 Mongo 状态；需要人工调用 `reindex` 或等待 Worker 重试
