# 动手实践 —— 更换 LLM 模型

Synapse 默认使用 Ollama 本地运行 `qwen2.5:7b`。本节讲解如何切换到其他模型，以及更换 Embedding 模型的注意事项。

## 修改对话模型

### 1. 修改 application.yaml

```yaml
synapse:
  llm:
    model-name: qwen2.5:7b    # ← 改成你想要的模型
```

支持的模型取决于你的 Ollama 已下载的模型：

```bash
ollama list
# NAME              ID            SIZE
# qwen2.5:7b        xxx           4.7 GB
# llama3.1:8b       xxx           4.9 GB
# mistral:7b        xxx           4.1 GB
```

### 2. 拉取新模型（如果本地没有）

```bash
ollama pull llama3.1:8b
```

### 3. 重启后端

```bash
# 重新编译启动
mvn spring-boot:run -pl synapse-kb-bootstrap
```

### 4. 验证

在聊天框提问，观察：
- 回答风格是否变化（不同模型的语气差异很大）
- 响应速度（大模型可能更慢）
- 中文能力（部分模型中文较弱）

## 更换 Embedding 模型（⚠️ 必须重建索引）

Embedding 模型决定文本如何变成向量。**不同模型的输出维度不同**，必须重建 Milvus Collection。

### 1. 查看当前配置

```yaml
synapse:
  embedding:
    model-name: gme-Qwen2-VL-2B-Instruct-GGUF:Q8_0
    dimension: 1536
```

### 2. 选择新模型并确认维度

常见 Embedding 模型维度：

| 模型 | 维度 | 说明 |
|------|------|------|
| nomic-embed-text | 768 | 轻量，英文为主 |
| mxbai-embed-large | 1024 | 多语言 |
| bge-m3 | 1024 | 中文优化 |
| gme-Qwen2 | 1536 | 当前使用，多模态 |

### 3. 拉取新模型

```bash
ollama pull nomic-embed-text
```

### 4. 修改配置

```yaml
synapse:
  embedding:
    model-name: nomic-embed-text
    dimension: 768          # ← 维度必须匹配！
```

### 5. 重建 Milvus Collection

**必须执行！** 因为向量维度变了，旧数据无法兼容。

```bash
# 方案一：用 Attu 图形界面删除 Collection
# 1. 打开 Attu → Collections → 删除 document_chunks
# 2. 重启后端，自动重新创建 Collection

# 方案二：代码中自动处理（如果已实现）
# 在 MilvusVectorStoreAdapter 初始化时检查维度，不匹配则删除重建
```

### 6. 重新上传所有文档

Collection 清空后，所有已有文档需要重新上传并摄入。

## 为什么维度必须匹配

Milvus 的 Collection Schema 在创建时就固定了向量字段的维度：

```java
FieldType vectorField = FieldType.newBuilder()
    .withName("vector")
    .withDataType(DataType.FloatVector)
    .withDimension(1536)   // ← 创建后不可修改
    .build();
```

尝试写入 768 维向量到 1536 维的字段会直接报错。

## 性能对比参考

| 模型 | 维度 | 本地 8C CPU 速度 | 中文质量 |
|------|------|----------------|---------|
| nomic-embed-text | 768 | 很快 | 中等 |
| bge-m3 | 1024 | 中等 | 很好 |
| gme-Qwen2 | 1536 | 较慢 | 很好 |

维度越低，Milvus 搜索越快、占用磁盘越少，但语义表达能力可能稍弱。
