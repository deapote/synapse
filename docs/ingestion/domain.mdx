### Document.status —— 摄入状态机

`Document.status` 描述文档在摄入 pipeline 中的处理状态，与业务时效状态严格分离。

| 当前状态 | 允许转换到 | 说明 |
|---------|-----------|------|
| PENDING | PROCESSING | 开始处理 |
| PROCESSING | COMPLETED | 处理成功 |
| PROCESSING | FAILED | 处理失败 |
| FAILED | PENDING | 重试 |
| COMPLETED | （无） | 终态 |

```java
private static final Map<DocumentStatus, Set<DocumentStatus>> VALID_TRANSITIONS = Map.of(
    DocumentStatus.PENDING,     EnumSet.of(DocumentStatus.PROCESSING),
    DocumentStatus.PROCESSING,  EnumSet.of(DocumentStatus.COMPLETED, DocumentStatus.FAILED),
    DocumentStatus.FAILED,      EnumSet.of(DocumentStatus.PENDING),
    DocumentStatus.COMPLETED,   EnumSet.noneOf(DocumentStatus.class)
);
```

### DocumentLifecycleStatus —— 业务时效状态

文件：`synapse-kb-domain/.../model/DocumentLifecycleStatus.java`

```java
public enum DocumentLifecycleStatus {
    ACTIVE,      // 生效中
    SUPERSEDED,  // 已被新版本替代
    RETIRED      // 已废止
}
```

`DocumentLifecycleStatus` 与 `DocumentStatus` 的关系：

- `DocumentStatus` 管**摄入过程**（文件是否已解析、分块、索引）。
- `DocumentLifecycleStatus` 管**业务时效**（这份资料在当前查询时间点是否有效）。
- 一个 `COMPLETED` 的文档可以是 `ACTIVE`，也可以是 `SUPERSEDED` 或 `RETIRED`。
- 一个 `FAILED` 的文档没有检索价值，其 `lifecycleStatus` 理论上不影响检索（因为 `FAILED` 不会写入索引）。

### DocumentMetadata —— 时效元数据值对象

文件：`synapse-kb-domain/.../model/DocumentMetadata.java`

```java
public record DocumentMetadata(
    DocumentSourceType sourceType,      // LEGAL, POLICY, GENERAL
    String canonicalKey,                // 规范标识，多版本关联键
    String versionLabel,                // 版本标签，如 "2024 版"
    LocalDate effectiveFrom,            // 生效日期
    LocalDate effectiveTo,              // 排他结束日期
    String supersedesDocumentId,        // 被替代的旧文档 ID
    Integer authorityLevel,             // 权威等级，数值越高优先级越高

```

所有字段均可空，由 domain 层提供默认值：
- `sourceType` 默认 `GENERAL`
- `effectiveFrom` 默认上传日期
- `lifecycleStatus` 默认 `ACTIVE`
- `authorityLevel` 默认 `0`

### effectiveTo —— 排他结束日

`effectiveTo` 的语义是**排他**的：

| effectiveTo | 含义 |
|-------------|------|
| `null` | 永久有效 |
| `2025-01-01` | 2025-01-01 **当天起无效** |

这意味着如果旧法规的 `effectiveTo=2025-01-01`，查询 `asOfDate=2025-01-01` 时不会命中该法规。新法规的 `effectiveFrom=2025-01-01` 才会被命中。

### isEffectiveOn() —— 时效校验

```java
public boolean isEffectiveOn(LocalDate date) {
    if (date == null) return false;
    if (lifecycleStatus == DocumentLifecycleStatus.RETIRED) return false;
    if (effectiveFrom != null && date.isBefore(effectiveFrom)) return false;
    if (effectiveTo != null && !date.isBefore(effectiveTo)) return false;
    return true;
}
```

### supersedeBy() —— 版本替代

```java
public void supersedeBy(Document newDocument, LocalDate effectiveTo) {
    if (lifecycleStatus != DocumentLifecycleStatus.ACTIVE) {
        throw new DomainException("仅 ACTIVE 文档可被替代");
    }
    if (effectiveTo == null) {
        throw new DomainException("替代失效日期不能为空");
    }
    if (this.effectiveFrom != null && effectiveTo.isBefore(this.effectiveFrom)) {
        throw new DomainException("替代失效日期不得早于文档生效日期");
    }
    this.effectiveTo = effectiveTo;
    this.lifecycleStatus = DocumentLifecycleStatus.SUPERSEDED;
}
```

**业务语义**：旧文档被新版本替代时，旧文档的 `effectiveTo` 被设为新版本的 `effectiveFrom`，`lifecycleStatus` 变为 `SUPERSEDED`。旧文档本身不删除，索引中的 scalar 字段保持不变（检索时由 application 层用 `isEffectiveOn()` 兜底过滤）。

### create() 工厂方法

```java
public static Document create(KnowledgeBaseId knowledgeBaseId, String fileName,
                               String fileType, long fileSize, String contentHash,
                               DocumentMetadata metadata) {
    if (fileName == null || fileName.isBlank()) {
        throw new DomainException("文件名不能为空");
    }
    Document doc = new Document(DocumentId.generate(), ...);
    if (metadata != null) {
        doc.applyMetadata(metadata);
    }
    return doc;
}
```

**强制 PENDING**：新创建的文档一定是 PENDING 状态，`lifecycleStatus` 默认 `ACTIVE`。

    private Instant updatedAt;
}
```

IngestionJob 有自己的状态机：

```
QUEUED → RUNNING → SUCCEEDED
              ↓
           RETRYING → QUEUED
              ↓
           FAILED
```

### 租约机制

```java
public void claim(String workerId, Duration leaseDuration) {
    this.leaseOwner = workerId;
    this.leaseExpiresAt = Instant.now().plus(leaseDuration);
    this.status = IngestionJobStatus.RUNNING;
}

public boolean isLeaseExpired() {
    return leaseExpiresAt != null && Instant.now().isAfter(leaseExpiresAt);
}
```

**租约机制的作用**

假设 Worker A 认领了任务，开始处理。处理过程中 Worker A 崩溃了，任务一直卡在 RUNNING 状态。其他 Worker 看到 RUNNING 状态就不会处理它，任务永远完不成。

租约机制：Worker A 认领时设置 `leaseExpiresAt = now() + 5分钟`。如果 5 分钟后 Worker A 还没完成，租约过期，其他 Worker 可以重新认领。

## RecursiveChunkingStrategy —— 文本分块

文件：`synapse-kb-domain/.../service/RecursiveChunkingStrategy.java`

```java
public class RecursiveChunkingStrategy {
    private final int maxSegmentSize;   // 最大分块大小（默认 1000 字符）
    private final int overlapSize;      // 重叠窗口大小

    public List<DocumentChunk> split(String text) {
        List<DocumentChunk> chunks = new ArrayList<>();
        int position = 0;

        while (position < text.length()) {
            int end = Math.min(position + maxSegmentSize, text.length());
            int breakPoint = findBreakPoint(text, position, end);

            String chunkText = text.substring(position, breakPoint).strip();
            if (!chunkText.isEmpty()) {
                chunks.add(new DocumentChunk(index, chunkText, position, breakPoint));
            }
            position = findOverlapStart(text, position, breakPoint);
        }
        return chunks;
    }
}
```

### 分块边界优先级

`findBreakPoint()` 按优先级寻找最佳切分点：

1. **Markdown 边界**（标题 `#`、列表 `-`、代码块 ` ``` `）
2. **段落边界**（`\n\n`）
3. **句子边界**（句号、问号、感叹号）
4. **软边界**（空格、标点）

### 分块重叠的设计原因

```
[段落1的后半部分] [段落2的前半部分]
        ↑ 重叠区域 ↑
```

如果句子跨越两个块的边界，没有重叠的话这个句子会被切断，语义丢失。重叠窗口（默认 15%）保证每个块的开头包含上一个块结尾的部分内容，减少信息损失。
