# 上传流程完整链路

PDF 从浏览器上传到可被提问检索的完整链路，涉及 Controller 接收、查重、解析、分块、向量化、存储。

## 九步链路

### 1. 接收上传请求

**Adapter · DocumentController**

HTTP 请求进入 `DocumentController.upload()`，返回 `Mono<DocumentResponse>`。

```java
DataBufferUtils.join(file.content())  // 将上传片段拼成完整 byte[]
```

### 2. 构建入站命令

**Adapter · DocumentController**

生成 SHA-256 哈希用于去重，提取 Content-Type，封装为 `IngestDocumentCommand`。

```java
.subscribeOn(Schedulers.boundedElastic())  // 保证阻塞操作不影响主线程
```

### 3. 查重与文档创建

**Application · KnowledgeBaseApplicationService.ingest()**

用例编排：查重 → 决定拒绝、清理重试、或全新创建。

1. **查重：** `findByKnowledgeBaseIdAndContentHash(kbId, hash)`
2. **COMPLETED/PROCESSING：** 抛异常拒绝
3. **FAILED：** 清理残留后允许重试
4. **全新文档：** `Document.create(...)`，状态自动为 PENDING

### 4. 持久化文档元数据

**Adapter · MongoDocumentRepository**

领域实体 `Document` 转换为 MongoDB 文档实体 `DocumentDocument`。

```java
documentMongoRepository.save(docEntity).block()  // .block() 将 Reactive 桥接为同步 API
```

### 5. 文档内容处理

**Application · processDocument()**

状态流转 → 解析 → 分块 → 向量化 → 存储，全流程串行编排。

1. `document.transitionTo(PROCESSING)`
2. **解析：** `documentParserPort.parse(content, fileName)` → 纯文本
3. **分块：** `recursiveChunkingStrategy.split(text)` → List<DocumentChunk>
4. **向量化：** `embeddingPort.embed(chunkTexts)`
5. **存储：** `vectorStorePort.store(kbId, documentId, chunks, embeddings)`

### 6. 文本分块策略

**Domain · RecursiveChunkingStrategy**

纯 Java 实现，零框架依赖。按字符切分，边界优先段落结束、句子结束，块间保留重叠。

每个 `DocumentChunk` 包含 index、text、startPosition、endPosition。

### 7. 向量生成

**Adapter · OllamaEmbeddingAdapter**

调用 Ollama Embedding 模型（gme-Qwen2-VL-2B，1536 维），批量将文本转为向量。

```java
Schedulers.boundedElastic()  // 隔离阻塞的 Ollama 网络调用
```

### 8. 向量存储

**Adapter · MilvusVectorStoreAdapter**

文档块及向量存入 Milvus，collection 为 `synapse_document_chunks`。

- 索引：IVF_FLAT
- 度量：COSINE
- 维度：1536
- 必须传入 knowledgeBaseId

### 9. 返回响应

**Adapter · DocumentController**

`ingest()` 返回 `DocumentId`，Controller 包装为 `Mono<DocumentResponse>` 返回前端。

## 数据最终去向

| 数据类型 | 存储位置 | 技术组件 |
|---------|---------|---------|
| 文档元数据（id, name, status, hash） | MongoDB | `MongoDocumentRepository` |
| 文档分块 + 向量 | Milvus | `MilvusVectorStoreAdapter` |
| 领域对象（运行时） | 内存 | `Document`, `DocumentChunk` |

## 查重与状态机

### 为什么用 contentHash 不用 fileName？

| 场景 | 用 fileName | 用 contentHash |
|------|------------|---------------|
| 同一文件改个名字再传 | 重复入库 | 正确拒绝 |
| 不同文件恰好同名 | 错误拒绝 | 正确允许 |
| 文件内容修改后再传 | 允许 | 允许（哈希变了） |

### 三种状态的处理逻辑

| 状态 | 处理 |
|------|------|
| **COMPLETED** | 直接拒绝。已完整解析、分块、向量化、入库 |
| **PROCESSING** | 直接拒绝。防止并行处理造成重复向量 |
| **FAILED** | 清理残留后允许重试。先删向量再删元数据 |

<Danger>
  清理顺序不能反：必须先删向量（Milvus），再删元数据（MongoDB）。如果先删元数据，向量残留就变成"无头向量"——永远清理不掉，还可能在检索时干扰结果。
</Danger>
