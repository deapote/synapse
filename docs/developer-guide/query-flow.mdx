# 问答检索完整链路

用户提问后，后端从问题向量化到 SSE 流式输出的完整链路。

## RAG 核心概念

不是直接问 LLM，而是先从知识库找相关内容，把找到的内容和用户问题一起喂给 LLM。LLM 基于检索到的片段生成答案，而不是凭记忆回答。

## 完整链路

### 1. 接收提问

**Adapter · StreamingQueryController**

接收 `POST /api/knowledge-bases/{kbId}/query/stream`，返回 `Flux<ServerSentEvent<String>>`。

### 2. 非阻塞包装

**Adapter · StreamingQueryController**

`Mono.fromCallable + Schedulers.boundedElastic()` 包装同步调用，防止阻塞 Netty 事件循环。

### 3. 用例编排

**Application · QueryKnowledgeBaseUseCase.prepare(query)**

协调检索、构建 prompt、调用 LLM。

### 4. Query 改写与质量门禁

**Adapter · OllamaQueryRewriteAdapter**

尝试改写用户问题以获得更好的检索效果，然后用 embedding 余弦相似度做质量门禁（默认阈值 `0.8`）。未通过时自动回退原 query。

### 5. 问题向量化

**Adapter · OllamaEmbeddingAdapter**

把用户问题转成 1536 维向量，和文档向量用同一个模型。

### 6. 混合检索

**Adapter · MilvusVectorStoreAdapter + MongoChunkSearchIndexAdapter**

并行执行两种检索：

- **向量召回（Milvus）**：基于语义相似度，找 Top-K 最相似的文档片段
- **关键词召回（MongoDB BM25）**：基于关键词匹配

然后融合重排，得到最终检索结果。

### 7. Prompt 构建

**Application · Prompt 组装逻辑**

把检索到的所有 Top-K 片段和用户问题拼接成完整 prompt：

```
基于以下参考资料回答用户问题：

[1] 片段内容 A...
[2] 片段内容 B...
[3] 片段内容 C...

用户问题：{question}

请基于以上参考资料回答，如果资料不足以回答，请准确说明。
```

### 8. 封装检索上下文

**Domain · RagContext 值对象**

`RagContext` 封装 prompt + 引用来源列表（`List<ChunkReference>`），供 LLM 生成和前端展示引用。

### 9. LLM 流式生成

**Adapter · OllamaStreamingLlmAdapter**

调用 Ollama（qwen2.5:7b），SSE 流式返回生成的文本。

### 10. SSE 推送给前端

**Adapter · StreamingQueryController**

把 LLM 的流式输出包装成 `ServerSentEvent`，逐字推送给前端，实现打字机效果。

## 为什么必须用 SSE？

LLM 生成答案很慢（几秒到十几秒）。如果等全部生成完再返回，用户对着空白页面干等。

- **SSE**：后端生成一个字，前端立刻显示一个字
- **单向流**：后端 → 前端，前端不能通过 SSE 发数据
- **基于 HTTP**：比 WebSocket 轻量，无需握手

## 核心安全边界：禁止跨知识库查询

<Danger>
  为什么 VectorStorePort.search() 必须传入 knowledgeBaseId？

  上传时：`vectorStorePort.store(kbId, documentId, chunks, embeddings)` —— 向量带上了知识库 ID。
  查询时：`vectorStorePort.search(kbId, vector, topK)` —— 只在指定知识库内检索。

  如果不传 kbId，用户在知识库 B 提问时可能检索到知识库 A 的文档，造成数据泄露。
</Danger>

## 常见误区

| 误区 | 正确理解 |
|------|---------|
| 只取第一个检索结果给 LLM | 错误。RAG 的核心是把 Top-K 所有片段一起给 LLM 当上下文 |
| 检索结果"也给前端看看" | 模糊。检索结果有两个用途：给 LLM 当上下文 + 给前端当引用来源 |
| Prompt 构建是技术细节 | 错误。Prompt 怎么拼是业务决策，属于 Application 层 |

## Prompt 构建细节

KnowledgeBaseApplicationService.prepare() 的 5 个步骤：

1. **问题向量化** —— `embeddingPort.embed(query.text())`
2. **向量检索 Top-K** —— `vectorStorePort.search(kbId, embedding, topK)`
3. **格式化上下文 + 构建引用** —— 每条片段格式化为 `[1] 片段文本\n\n`
4. **组装 Prompt** —— `String.format(promptTemplate, context, query)`
5. **返回 RagContext** —— 值对象封装 prompt 和引用来源

<Tip>
  每个检索片段分配的 `sourceId` 有两个作用：
  1. 给 LLM 做引用标记，回答时可引用 `[1]` 表示来自第一个片段
  2. 给前端做来源展示，用户可点击序号查看原始文档位置
</Tip>
