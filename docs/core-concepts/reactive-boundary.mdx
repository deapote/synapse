# Reactive 边界设计

## 核心决策

Synapse 使用 Spring WebFlux 作为 Web 层框架，但 Application 层和 Domain 层完全保持同步。这是一个**有意的架构决策**。

## 为什么 Application 层是同步的？

业务逻辑不该知道 HTTP 请求是怎么发的、数据库是不是 reactive 的。它只关心：

- 查重规则是什么？
- 状态怎么流转？
- 分块策略怎么选？

这些跟 I/O 无关，所以应该是同步的。Reactive 是技术实现层面的选择，不是业务层面的选择。

## 五个原因

### 1. 零框架依赖

Domain 层禁止引入 Spring、Reactor。如果 Application 返回 `Mono`，Domain 就被迫理解 `Mono`。

### 2. 业务是决策流

查重 → 创建 → 保存 → 解析 → 分块 → 向量化 → 存储

严格串行，没有并行必要。

### 3. Reactive 解决 I/O

Controller、MongoDB、Ollama 都是 I/O。这些是 Adapter 的技术细节，Application 不该知道。

### 4. 可测试性

同步 API 的单元测试干净直接：

```java
@Test
void shouldRejectDuplicateDocument() {
    // given
    var command = new IngestDocumentCommand(kbId, "file.pdf", hash, content);
    when(documentRepo.findByKnowledgeBaseIdAndContentHash(kbId, hash))
        .thenReturn(List.of(existingCompletedDoc));

    // when/then
    assertThrows(DomainException.class, () -> service.ingest(command));
}
```

如果用 Mono，需要 `StepVerifier`、调度器——复杂度翻倍。

### 5. 依赖方向保护

```
domain ← application ← adapter
```

Application 依赖 Reactor 会破坏向内指向 domain 的规则。

## .block() 是刻意的边界

```java
// Adapter 层消化技术复杂性
DocumentDocument docEntity = documentMongoRepository.save(entity).block();

// Application 层只面对纯净 Java 对象
Document document = DocumentMapper.toDomain(docEntity);
```

`Mono<DocumentDocument> → .block() → Document`。Adapter 负责把 reactive 世界桥接到同步的业务世界。

## 本项目的 Reactive 边界

```
┌─────────────────────────────────────────────────────┐
│  Adapter In (WebFlux)                                │
│  Controller 返回 Mono<> / Flux<>                      │
│  线程不阻塞，处理 HTTP I/O                             │
├─────────────────────────────────────────────────────┤
│  Application (同步)                                   │
│  业务逻辑纯净，只操作 Java 对象                         │
├─────────────────────────────────────────────────────┤
│  Adapter Out (Mono/Flux + .block())                  │
│  MongoDB、Ollama、Milvus 调用                        │
│  .block() 将 reactive 桥接为同步                      │
└─────────────────────────────────────────────────────┘
```

## Mono & Flux

| 类型 | 含义 | 场景 |
|------|------|------|
| `Mono<T>` | 0 或 1 个结果的异步操作 | 查一条记录、保存一个文档 |
| `Flux<T>` | 0 到 N 个结果的异步流 | 查列表、SSE 流式输出 |

<Info>
  Mono 和 Flux 不是数据，是**"数据的承诺"**。订阅之前什么都不会发生。
</Info>

## SSE 流式输出的特殊处理

问答接口使用 `Flux<ServerSentEvent<String>>` 返回流式输出：

```java
@PostMapping(value = "/query/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> streamQuery(
        @PathVariable String kbId,
        @RequestBody QueryRequest request) {
    return Mono.fromCallable(() -> {
        // 同步业务逻辑
        RagContext context = queryUseCase.prepare(new Query(kbId, request.query()));
        return context;
    })
    .subscribeOn(Schedulers.boundedElastic())
    .flatMapMany(context -> {
        // 流式输出
        return streamingLlmPort.generateStream(context.prompt())
            .map(token -> ServerSentEvent.builder(token).event("token").build());
    });
}
```

<Tip>
  `Mono.fromCallable + Schedulers.boundedElastic()` 包装同步调用，防止阻塞 Netty 事件循环。
</Tip>
