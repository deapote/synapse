# 文本分块策略

## 为什么需要分块

大语言模型有上下文长度限制，无法一次性处理整篇文档。RAG 系统将文档切分成小块，每块独立向量化和检索，再把检索到的相关片段作为上下文提供给 LLM。

## RecursiveChunkingStrategy

`RecursiveChunkingStrategy` 是 Domain 层的纯 Java 实现，零框架依赖。它按字符切分，边界优先选择语义完整的分割点。

### 边界优先级

```
Markdown 标题分割 > 段落结束 > 中英文句子结束 > 软标点（逗号、顿号） > 硬切（强制字符数）
```

### 核心参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `chunkSize` | 1000 | 每块目标字符数 |
| `chunkOverlap` | ~15% | 块间重叠窗口，保障语义连续性 |

<Tip>
  重叠窗口配置范围限制在 10% 到 20%。太小会丢失跨块语义，太大会增加冗余存储。
</Tip>

### 分块结果

每个 `DocumentChunk` 包含：

```java
public record DocumentChunk(
    int index,          // 块序号
    String text,        // 块文本内容
    int startPosition,  // 在原文中的起始位置
    int endPosition     // 在原文中的结束位置
) {}
```

## 分块示例

假设输入文本：

```
# 第一章 引言

人工智能正在改变我们的生活。从语音识别到自动驾驶，AI 技术无处不在。

## 1.1 背景

深度学习的发展推动了自然语言处理的突破。Transformer 架构的出现...
```

分块结果（假设 chunkSize=100）：

| 序号 | 文本内容 | 起始位置 |
|------|---------|---------|
| 0 | `# 第一章 引言\n\n人工智能正在改变我们的生活。从语音识别到自动驾驶...` | 0 |
| 1 | `AI 技术无处不在。\n\n## 1.1 背景\n\n深度学习的发展推动了...` | ~80 |

注意第二块的开头包含了第一块末尾的部分内容（重叠窗口），保证跨块语义的连续性。

## 为什么放在 Domain 层？

分块策略是**业务规则**的一部分：

- 选择什么样的边界？—— 段落优先还是句子优先？
- 重叠多少？—— 10% 还是 20%？
- 最大块多大？—— 800 字还是 1200 字？

这些决策取决于业务场景（法律文档 vs 技术文档 vs 聊天记录），不是技术细节。因此放在 Domain 层，由领域专家决定。

## 与 LangChain4j 的关系

Synapse 的 `RecursiveChunkingStrategy` 是自主实现的 Domain Service，不依赖 LangChain4j。Adapter 层的 `ApacheTikaDocumentParserAdapter` 负责调用 Tika 解析文档为纯文本，然后将文本交给 Domain 层的分块策略处理。

这种设计保持了 Domain 层的纯净：即使将来替换 Tika 或 LangChain4j，分块逻辑也不需要改动。
