# DDD 分层架构

## 为什么 Application 层是同步的？

Adapter 层用 WebFlux，Application 层却返回普通对象。这是刻意的架构决策。

> 业务逻辑不该知道 HTTP 请求是怎么发的、数据库是不是 reactive 的。它只关心"查重规则是什么"、"状态怎么流转"。这些跟 I/O 无关，所以应该是同步的。Reactive 是技术实现层面的选择，不是业务层面的选择。

### 五个原因

| 原因 | 说明 |
|------|------|
| **1. 零框架依赖** | Domain 层禁止引入 Spring、Reactor。如果 Application 返回 Mono，Domain 就被迫理解 Mono |
| **2. 业务是决策流** | 查重 → 创建 → 保存 → 解析 → 分块 → 向量化 → 存储。严格串行，没有并行必要 |
| **3. Reactive 解决 I/O** | Controller、MongoDB、Ollama 都是 I/O。这些是 Adapter 的技术细节，Application 不该知道 |
| **4. 可测试性** | 同步 API 的单元测试干净直接。Mono 需要 StepVerifier、调度器——复杂度翻倍 |
| **5. 依赖方向保护** | domain ← application ← adapter。Application 依赖 Reactor 会破坏向内指向 domain 的规则 |

<Tip>
  `Mono<DocumentDocument> → .block() → Document`。Adapter 消化技术复杂性，Application 只面对纯净 Java 对象。
</Tip>

## Reactive 的本质

### 阻塞 vs 非阻塞

| 模式 | 类比 | 特点 |
|------|------|------|
| **阻塞（Spring MVC）** | 点完餐站在柜台前干等 | 线程等待 I/O，什么事都做不了 |
| **非阻塞（WebFlux）** | 点完餐拿叫号器去座位等 | 线程可以处理其他请求，数据好了通知你 |

### Mono & Flux

| 类型 | 含义 | 场景 |
|------|------|------|
| `Mono<T>` | 0 或 1 个结果的异步操作 | 查一条记录、保存一个文档 |
| `Flux<T>` | 0 到 N 个结果的异步流 | 查列表、SSE 流式输出 |

<Info>
  Mono 和 Flux 不是数据，是**"数据的承诺"**。订阅之前什么都不会发生。
</Info>

### 本项目的 Reactive 边界

```
[浏览器] ──HTTP──► [Controller: Mono/Flux]      ←── 线程不阻塞
                           │
                           ▼
                [Application Service: 同步]         ←── 业务逻辑纯净
                           │
                           ▼
                [Adapter Out: Mono/Flux]            ←── MongoDB、Ollama
                           │
                           ▼
                     .block()                        ←── 桥接
```

## Application Service vs Domain Service

| 特征 | Domain Service<br/>(如 `RecursiveChunkingStrategy`) | Application Service<br/>(如 `KnowledgeBaseApplicationService`) |
|------|---------------------------------------------------|---------------------------------------------------------------|
| **所在层** | Domain | Application |
| **有状态吗** | 无字段 | 无字段 |
| **依赖什么** | 只依赖领域对象 | 依赖端口接口（Repository、Port） |
| **知道技术细节吗** | 不知道 | 知道端口存在，不知道具体实现 |
| **决策依据** | 业务规则（段落优先、重叠长度） | 用例步骤顺序（查重→解析→分块） |

> **核心判断标准：** 领域对象只负责自己"知道"的事情。Document 不知道 text，所以不应该由它来校验 text 的长度。`RecursiveChunkingStrategy` 无状态、无身份，只是一个"算法容器"，操作的是传入的数据而不是自己的数据。

## Command vs Value Object

| | `IngestDocumentCommand` | `DocumentContent` |
|---|------------------------|-------------------|
| **所在层** | Application | Domain |
| **含技术类型** | InputStream（I/O 流） | 只有 String |
| **用例特定** | 只用于"上传文档" | 可跨用例复用 |
| **本质** | "用户想执行某个动作" | "领域中的一个概念" |

## 构造函数 vs reconstruct

| | 构造函数 | reconstruct |
|---|---------|-------------|
| **调用方** | Application Service | Repository |
| **场景** | 用户上传新文档（"生"） | 从数据库恢复（"还魂"） |
| **状态** | 强制 PENDING | 数据库里是什么就是什么 |
| **时间戳** | createdAt = now() | createdAt = 数据库旧值 |
| **业务规则** | 强制执行 | 必须绕过 |
| **为什么 static** | — | 调用时还没有实例 |

<Warning>
  为什么 reconstruct 必须放在 Document 类内部？因为 reconstruct 需要访问 Document 的 private 字段来还原状态。如果放在 Repository 里，无法访问私有字段，只能走构造函数——但构造函数会重置状态和时间戳。
</Warning>

> **一句话总结：** 构造函数管"出生"，强制执行所有业务规则。reconstruct 管"复活"，完整还原历史状态，不碰任何规则。

## 分层对照表

| 层级 | 模块 | 职责 | 同步/Reactive | 框架依赖 |
|------|------|------|--------------|---------|
| **Domain** | synapse-kb-domain | 实体、值对象、领域服务、仓储接口 | 同步 | 零框架依赖 |
| **Application** | synapse-kb-application | 用例编排、端口定义 | 同步 | 无 Spring/Reactor |
| **Adapter** | synapse-kb-adapter | Controller、Repository 实现、LLM 调用 | Reactive（.block() 桥接） | Spring、LangChain4j、Reactor |
| **Config** | synapse-kb-config | Bean 组装、依赖注入 | — | Spring |
